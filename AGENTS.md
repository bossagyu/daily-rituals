# エージェント構成

## ロール一覧

| ロール | ファイル | 責務 |
|--------|----------|------|
| PO | `.claude/agents/po.md` | 要件定義、issue作成、優先順位付け、タスクアサイン |
| dev | `.claude/agents/dev.md` | TDDで実装、PR作成、レビュー対応 |
| DA | `.claude/agents/da.md` | 批判的レビュー、品質検証、エスカレーション |
| TL | `.claude/agents/tl.md` | 技術的最終判断、アーキテクチャ指針 |

## ワークフロー

```
PO: issue作成・アサイン
  → dev: 実装・PR作成
    → DA: レビュー
      → (合意) → マージ
      → (膠着) → TL: 最終判断
        → マージ
```

### 並行実装時のルール

複数issueを並行で実装する場合、以下のルールを適用する：

1. **共有ファイルの事前特定**: POはissue作成時に、複数issueが同一ファイルを変更する可能性がある場合、issueの技術的メモに明記する（例: `index.ts` のエクスポート追加）
2. **マージ順序の指定**: 並行PRが同一ファイルを変更する場合、POがマージ順序を指定する。後続PRはリベースしてからマージする
3. **コンフリクト防止パターン**: バレルファイル（`index.ts`）への追加が複数PRで発生する場合、各PRはそのモジュール固有のエクスポートのみを追加し、既存行を変更しない

## 運用上の注意事項

- 個人リポジトリではGitHubの制限により、自身のPRに対してAPPROVEを付与できない。DAのレビュー判定はCOMMENTとして投稿し、本文中にAPPROVE/REQUEST_CHANGESの判定を明記する運用とする。
- issueの受け入れ基準には、セキュリティ関連の要件（`.gitignore`設定、秘密情報の取り扱い）を具体的に含めること。曖昧な基準はレビュー段階での手戻りの原因になる。

## テスト戦略

### 統合テストの扱い

expo-sqliteはネイティブモジュールのため、Jest（Node.js）環境では直接利用できない。以下の方針で統合テストを実施する：

- **リポジトリ層の統合テスト**: `better-sqlite3` のインメモリDBでexpo-sqlite APIのアダプターを作成し、実際のSQLiteエンジンで検証する（Sprint 3で確立したパターン）
- **issueの受け入れ基準での明記**: 「統合テスト」を要求する場合は、テスト手法（`better-sqlite3`アダプター使用）も併せて記載する
- **将来**: E2Eテスト環境が整った段階で、実機でのSQLite操作を検証する

### テスト区分の定義

| 区分 | 対象 | 手法 |
|------|------|------|
| ユニットテスト | ドメインサービス、マッパー関数 | モックベース |
| 統合テスト | リポジトリ層 | `better-sqlite3` インメモリDB |
| E2Eテスト | ユーザーフロー全体 | 実機/エミュレータ（将来） |


## UI層のDI（依存性注入）戦略

UI層（スクリーン）がリポジトリ等の依存を取得する方法として、React Contextを使ったDIパターンを採用する。

- `RepositoryProvider`（`App.tsx`）がコンポーネントツリーのルートでリポジトリインスタンスを提供する
- 各スクリーンは `useRepositories()` フックを通じてリポジトリを取得する
- React Navigationのスクリーン定義ではpropsの直接注入が困難なため、Contextパターンが標準
- テスト時はProviderをモックすることで依存を差し替え可能

## 振り返りルール

### タイミング

以下のいずれかが完了した時点で振り返りを実施する：
- 1つのissueの実装が完了しマージされた時
- スプリント（一連のissue群）が完了した時
- 開発プロセス上の問題が発生した時

### 振り返りの実施者

**POが振り返りを主導する。** 実装完了後、POは以下を確認する：
1. 開発プロセスに改善点はあったか
2. エージェント間の連携に問題はなかったか
3. 各エージェントの役割定義に過不足はなかったか

### 振り返りで更新する対象

| ファイル | 更新内容 |
|----------|----------|
| `AGENTS.md` | ワークフローの改善、ロール定義の変更 |
| `.claude/agents/*.md` | 各エージェントの役割・手順・判断基準の改善 |

### 振り返りの形式

```markdown
## 振り返り: [issue番号 or スプリント名]

### うまくいったこと
- ...

### 改善が必要なこと
- ...

### エージェント定義の変更
- [変更対象]: [変更内容と理由]
```

### 変更のルール

- 振り返り結果はgitコミットとして記録する（`chore: エージェント定義の振り返り更新`）
- 小さな改善は即座に反映する
- 大きな役割変更はユーザーの承認を得てから反映する
- 変更理由を必ずコミットメッセージに含める
